Bugs Found & Fixed

Critical Route Ordering Bug — /archived unreachable

The GET /applications/archived route was defined AFTER /{app_id}, so FastAPI matched "archived" as an integer app_id and threw a 422 error. Fixed by moving archived routes before dynamic routes.

Critical Scheduler queried non-existent field

Application.followup_due_date doesn't exist on the model — this would crash at runtime. Fixed to query the FollowUp table correctly for unsent follow-ups.

Critical Missing exports in api.js

getArchivedApplications and restoreApplication were used in ArchivedApplications.jsx but never exported from api.js — would throw a runtime error.

Critical App.jsx never rendered the lists

The original App.jsx only rendered Dashboard which only showed the form. ApplicationList and ArchivedApplications were built but never connected to the UI.

Bug Hardcoded credentials in database.py

MySQL password jarvis was hardcoded. Fixed to use DATABASE_URL environment variable. Also migrated MySQL → PostgreSQL for free cloud hosting.

Bug Archive always stored "Application Withdrawn"

archive_application() hardcoded the status to "Application Withdrawn" regardless of actual status. Fixed to preserve the real application status at time of archiving.

Bug FollowUpOut.sent_at missing default

Pydantic schema had sent_at: Optional[datetime] without = None, causing validation errors when returning unsent follow-ups where sent_at is NULL.

Bug No 404s for notes/followups on invalid app IDs

Adding notes or follow-ups to a non-existent application silently failed or caused a DB constraint error. Fixed with explicit app existence check returning 404.

Improvement Scheduler started at module import time

The scheduler ran scheduler.start() at import time, which causes issues in production WSGI/ASGI environments. Fixed to start via FastAPI's @app.on_event("startup").

Improvement Hardcoded CORS & API base URL

Frontend used hardcoded http://127.0.0.1:8000 and backend only allowed localhost:5173. Both now read from environment variables for production.

Free Platforms Guide

🗄 Neon.tech DB — Recommended

Free PostgreSQL with 10GB storage. Serverless, auto-scales to zero.

  • Sign up at neon.tech
  • Create a new project → copy connection string
  • Paste as DATABASE_URL in Render
  • Free tier: 10GB, no credit card needed

🗄 Supabase DB — Alternative

Free PostgreSQL with 500MB storage + nice dashboard.

  • Sign up at supabase.com
  • New project → Settings → Database → Connection String
  • Use "URI" format for SQLAlchemy
  • Free tier: 500MB, pauses after inactivity

⚙️ Render.com Backend

Free Python/FastAPI hosting. Sleeps after 15min inactivity (free tier).

  • Sign up at render.com
  • New Web Service → connect GitHub repo
  • Root dir: backend/
  • Build: pip install -r requirements.txt
  • Start: uvicorn main:app --host 0.0.0.0 --port $PORT
  • Add DATABASE_URL + ALLOWED_ORIGINS as env vars

⚡ Vercel Frontend

Best free React/Vite hosting. Instant deploys from GitHub.

  • Sign up at vercel.com
  • Import GitHub repo → set root to frontend/
  • Framework: Vite
  • Add env var: VITE_API_URL = your Render URL
  • Deploy — get a *.vercel.app URL instantly

Step-by-Step Deployment
1

Push your code to GitHub

Create a new GitHub repo and push the app-tracker/ folder. Both Vercel and Render will deploy from there automatically.

# In the app-tracker folder: git init git add . git commit -m "initial commit" git remote add origin https://github.com/YOUR_USERNAME/app-tracker.git git push -u origin main
2

Create free PostgreSQL on Neon.tech

Go to neon.tech → New Project → copy the connection string. It looks like:

postgresql://user:pass@ep-xxx.us-east-2.aws.neon.tech/neondb?sslmode=require

Save this — you'll paste it in Render next.

3

Deploy Backend on Render.com

Go to render.com → New Web Service → connect your GitHub repo.

Root Directory: backend Build Command: pip install -r requirements.txt Start Command: uvicorn main:app --host 0.0.0.0 --port $PORT # Environment Variables to add: DATABASE_URL = postgresql://... (paste Neon URL here) ALLOWED_ORIGINS = https://your-app.vercel.app

After deploy, note your Render URL (e.g. https://app-tracker-api.onrender.com)

4

Deploy Frontend on Vercel

Go to vercel.com → Import Git Repository → select your repo.

Root Directory: frontend Framework Preset: Vite # Environment Variable to add: VITE_API_URL = https://app-tracker-api.onrender.com
5

Update CORS on Render

After Vercel gives you your final URL (e.g. https://app-tracker.vercel.app), go back to Render → Environment → update ALLOWED_ORIGINS to match exactly.

ALLOWED_ORIGINS = https://app-tracker.vercel.app
